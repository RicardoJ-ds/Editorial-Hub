## Summary
We already ship a **schema** catalog into the Planning Hub (`gen_bq_schema_catalog.py` →
`editorial-team-pods/docs/bq_schema_catalog.{md,json}`), but it's *structural only* — name,
grain, columns, row count, freshness — and framed for the capacity consumer. We want a
**full ETL lineage catalog for ALL editorial tables/views (not just the capacity ones)**:
for each object, its **origin → processing → where the number surfaces in BOTH Hubs**.
And we want it **generated by you and auto-copied into `editorial-team-pods/docs/`** (same
sync mechanism as the schema catalog) so this repo is self-describing about the whole pipeline.

You own the ETL and the lineage knowledge (it's spread across `ETL_INVENTORY.md`,
`WAREHOUSE_DESIGN.md`, `handoff_planning_hub_capacity_data.md`, and the build/views code) —
so the ask is to consolidate it into one machine- + human-readable artifact and keep it fresh.

## What good looks like — a lineage catalog, one entry per table/view
Extend `gen_bq_schema_catalog.py` (or add a sibling `gen_bq_lineage_catalog.py`) that emits
`etl/bq_lineage_catalog.{md,json}` and **copies both into `editorial-team-pods/docs/`** the
same way. Per object, add these fields on top of what the schema catalog already has:

- **origin** — the true source. For `editorial_raw_*`: the source system + specific
  sheet / tab (e.g. "Editorial Tracker › Articles tab"). For `editorial_int_*` /
  `v_editorial_*`: the upstream table(s) it's derived from.
- **pipeline_step** — which stage/script produces it: `extract.py` → `transform.py` →
  `load.py` → `warehouse/build.py` (raw→int) → `views.py` (int→view). Name the function.
- **processing / business_math** — the transformation applied in plain language (weighting,
  variance, utilization, name-canonicalization, SF-identity join, collaboration explode, etc.).
  This is the part raw SQL doesn't reveal.
- **consumers.editorial_hub** — which Editorial-Hub page / endpoint / metric reads it.
- **consumers.planning_hub** — which editorial-team-pods reader reads it. For us the read
  layer is `src/lib/bq.ts` (BQ) and `src/lib/data.ts` (cached wrappers) → components. Name
  the function + the number it feeds (e.g. `getWriterDelivered()` → Writers "delivered").
- keep the existing **grain / columns / row_count / synced_at**.

A row should let a reader answer: *"this number on this screen came from that sheet, via
these steps, with this math."* — for every editorial object, both Hubs.

## Why we care (concrete drivers)
- The ETL will eventually **replace** the pod-assignment feed the Planning Hub writes to, so
  we need a shared, precise map of what feeds what before cutover.
- Debugging cross-Hub discrepancies (e.g. the roster-email issue in
  `handoff_planning_hub_roster_emails.md`) is slow without lineage — we had to reverse-engineer
  that email resolves as `COALESCE(dim_member.email, pod_assignments.email)`.
- We want this repo to be self-describing: a new engineer (or Claude session) opens
  `editorial-team-pods/docs/` and sees the whole pipeline, not just the schema.
