## Project quick-reference

- **Routes + folder structure**: `README.md`
- **Auth flow**: `src/lib/auth.ts`, `src/lib/session.ts`, `proxy.ts`
- **API client**: `src/lib/api.ts` (`apiGet`/`apiPost` against `NEXT_PUBLIC_API_URL`)
- **Shared types**: `src/lib/types.ts` — mirror of `backend/app/schemas.py`. When you add or rename a field in either place, update both.
- **Store pattern (CP v2)**: `src/app/(app)/capacity-planning/_store.tsx` is a React context backed by `localStorage`. Every CP v2 page reads + writes through this store. When the backend ships `/api/cp2/*` endpoints, swap `localStorage` calls for `apiGet`/`apiPost`; UI stays the same.
- **Filter bar**: `src/components/dashboard/FilterBar.tsx` fans out filtered client list + date range via `onFilterChange` + `onDateRangeChange`. Dashboards that want time-aware charts must consume both.
- **Pod label normalization**: `normalizePod()` appears in several files (`FilterBar.tsx`, `ContractClientProgress.tsx`, `DeliveryTrendChart.tsx`, `CumulativePipelineHeader.tsx`). "Pod 1" / "1" / "pod 1" all collapse to "Pod 1". Keep the function in sync across files or extract to `src/lib/pods.ts` next time someone touches it.
- **Pod display formatter**: `displayPod(pod, "editorial" | "growth")` in `src/components/dashboard/shared-helpers.tsx`. Returns `"Editorial Pod 1"` / `"Growth Pod 1"` / `"Unassigned"`. Keys passed around the code (`"Pod 1"` etc.) are never shown verbatim in user-facing copy — always wrap in `displayPod()` when rendering. `podBadge()` already does this internally.
- **Sync UX**: `SyncControls` (`src/components/layout/SyncControls.tsx`) holds the SYNC button + the "Synced Apr 23, 4:18 PM" badge. It's rendered inline with the filter row on D1/D2 (since `Header` returns null on those routes) and inside `Header` for every other route. Click → `SyncAllModal` → per-sheet fan-out to `/api/migrate/import` → synthetic **"Refresh computed KPIs"** step calling `POST /api/migrate/refresh-kpis` (recomputes the four Notion-derived KPIs from `notion_articles` + `capacity_projections` for every month with source data, capped at 36 months back) → final render is `SyncResultDetail`. One `window.dispatchEvent(new Event("data-synced"))` fires at the end; dashboards + the badge listen for this to refetch. The badge reads from `GET /api/migrate/status` (audit_log row written on every successful import) and renders the timestamp via `toLocaleString()` so it's automatically locale + timezone aware. Backend stores naive UTC datetimes — the helper in `SyncControls` appends `Z` if missing before parsing so we never drift by the user's UTC offset.
- **Team KPIs filter bar (`TeamKpiFilterBar`)**: matches D1's UX. Date period uses the same `DateRangeFilter` component (calendar + month-range slider + presets) — no separate Month/Year dropdowns. Member + Client are typeahead comboboxes via the local `FilterCombobox` helper (text input filters the dropdown, click commits, `X` clears). The page state is `dateRange` (not `month/year`); `/api/kpis/` is fetched with `year_from / month_from / year_to / month_to` and the heatmap aggregates per `(member × kpi_type)` using mean of non-null scores + the latest non-null target. Hovering any KPI column header surfaces a `TooltipBody` with source / formula / target / direction / **caveats** (paused upstream for AI Compliance, fallback heuristic for Second Reviews, pod-level replication for Capacity Utilization, status-name dependency for Revision Rate). Tooltip content lives in `KPI_HEATMAP_TOOLTIPS` at the top of `team-kpis/page.tsx` — keep that table the source of truth for caveats.
- **Section-header style**: Every top-level section on Dashboard 1 uses the same h2 pattern — `font-mono text-base font-bold uppercase tracking-[0.2em] text-white` + `h-px flex-1 bg-[#2a2a2a]` horizontal rule on the right. Card h3s use `text-sm font-semibold` (`#C4BCAA`). When adding a new section, mirror this to keep the visual hierarchy intact.
- **Sticky section headers + anchor nav**: each top-level `<section>` on D1/D2 carries `id="…"` + `scroll-mt-[140px]`, and the h2 wrapper is `sticky top-[120px] z-10 bg-black pb-2 pt-1`. The filter+tabs band that owns the top of the page has `sticky top-0 z-20` + `min-h-[120px]` so its `bg-black` butts up against the h2 sticky position with no transparent gap. Each tab also renders `SectionIndex` (`src/components/dashboard/SectionIndex.tsx`) — a sticky left-side rail with scroll-spy + click-to-jump (`topOffset = 140` matches `scroll-mt`). Critical: the page's scroller is the `overflow-auto` `<div>` from `app/(app)/layout.tsx`, not `window`. `SectionIndex` walks up the DOM and listens to every scrollable ancestor — keep it that way.
- **Scope-aware overview cards**: the top of each Deliverables-vs-SOW section renders one of three card sets depending on filter scope — single client / single pod / portfolio. Implementations: `DeliveryOverviewCards`, `GoalsOverviewCards`. Portfolio mode drops cumulative sums (sums across pods aren't actionable) and shows triage signals (closing-soon, last-Q closes). `MostBehindCard` and `PodAttentionCard` are exported from `DeliveryOverviewCards.tsx` but **not** mounted in D1 portfolio mode — they live exclusively on `/overview`. `framer-motion`'s `AnimatePresence` handles the layout swap when scope changes. The Cumulative Pipeline section instead uses an inline strip header (`CumulativePipelineHeader`): scope summary + 4 stage mini-bars + anomaly chip for portfolio/pod, status chip + pod chips + contract window for single-client. The per-client cards below carry the real detail.
- **Overview dashboard (`/overview`)**: exec snapshot. Page mounts `<TimeToMetrics>`, `<DeliveryOverviewCards>` (with lens switcher: composition / trajectory / triage), `<ProductionTrendChart>`, `<CumulativePipelineSection cardStages={["articles","published"]} defaultCollapsedByPod>`, `<ClientDeliveryCards defaultCollapsedByPod>`, and `<GoalsOverviewCards>`. Comments live per-section via `<OverviewCommentsProvider>` + `<SectionCommentIcon trailingSlot>` inline next to each section title (no rail anymore). Adds `<MostBehindCard>` + `<PodAttentionCard>` (kept off D1's portfolio scope). Lifetime delivery summaries come from `buildLifetimeSummaries()` in `src/lib/overviewSummary.ts`. Deep links use `?tab=deliverables-sow#section-id`. Page-level gate via `useRequireView("overview")` redirects pod-locked teams to `/editorial-clients`.
- **Cumulative Pipeline — Articles uses `articles_sent`**: Topics and CBs use the `approved` field from the Cumulative sheet because those gates require explicit client sign-off. Articles uses `articles_sent` because articles delivered to the client are billable regardless of the client's downstream approval workflow. Published uses `published_live`. Applies in `CumulativePipelineHeader.tsx` (the strip header on D1 + `/overview`) and `ClientPipelineCard.tsx` (the per-client cards below). The per-pod approval-rate matrix in `ContractClientProgress.tsx` (`PipelineCell`) intentionally still uses `articles_approved` — it measures approval rate (sent vs approved), a different question.
- **Pipeline stage palette**: `PIPELINE_STAGE_COLORS` in `src/components/dashboard/shared-helpers.tsx` — Topics #2E8C59 → CBs #42CA80 → Articles #65FFAA → Published #DDCFAC. Strictly Graphite DS swatches (P3 → P2 → P1 + WN1). Don't reintroduce S1/S2/S8 (blue/purple/cyan) on pipeline bars — flagged off-brand.
- **Pacing-aware coloring**: percentages that mean "% of contract progress" (e.g. lifetime delivered) use `pacingColor(actualPct, elapsedPct)` from `shared-helpers.tsx` — green inside ±10pp of expected, yellow at -25pp, red beyond. New clients (<8 % elapsed) always read green. Don't fall back to raw 75 / 50 thresholds for these metrics.
- **Content-type weighting**: goals aggregations multiply CBs / articles by `contentTypeRatio()` (article ×1, jumbo ×2, LP ×0.5). The content-type table is authoritative; the sheet's `ratios` column is fallback only because direction (`1:2` vs `2:1`) was inconsistent. Apply weighting consistently across `GoalsVsDeliverySection`, `aggregateGoalsByPod`, and `GoalsMonthTable` via 3-step aggregation (max-of-week per CMC → weighted client/month → pod totals) or totals diverge. Note this changed published numbers vs. pre-Apr 26 — a client with 17 article + 2 jumbo rows now reads 21, not 19.
- **"As of" labels** read from the Editorial week distribution (`editorial_weeks` table). `useEditorialAsOf()` returns the last fully-completed Editorial month (used by D1 / Overview headers + `ClientDeliveryCards`). `useCurrentEditorialMonth()` returns the in-progress month (used by Monthly Goals gauges in `GoalsVsDeliverySection`). When today lands outside imported weeks, AsOfBadge falls back to the calendar month + appends a `· cal.` chip. Both hooks live in `src/lib/editorialWeeksClient.tsx`.
- **RBAC + access gating**: every backend call carries `X-User-Email` (and `X-Preview-As` when admin impersonates) — see `src/lib/api.ts`. `useAccessProfile()` (`src/lib/accessClient.tsx`) reads `/api/access/me` once on mount, caches, and silently re-fetches on `visibilitychange` so other tabs see access changes without a manual reload. Consumers gate UI on `view_slugs`, `is_admin`, `pod_kind_lock`, `client_scope`. Sidebar filters nav items by `view_slugs`. New routes that join the matrix MUST add a row to the `_VIEWS` catalog in `backend/app/services/access.py` (5-tuple now: slug, label, dashboard_label, section, sort_order) AND map their slug in `Sidebar.tsx`'s `requiredViews`. Page-level gates: `useRequireView(slug)` redirects to `/editorial-clients` when access is missing.
- **Access Control matrix gating**: editing the cells is split into two tiers. `canEditCell` (Groups tab) and `canEditOverride` (Users × Views) live in `admin/access/page.tsx` — Admin-group rows are read-only, `admin.access` / `admin.access.edit` mutations are admin-only, seeded admins (Daniela / Ricardo) are immutable across the matrix. Backend mirrors all gates with 403s; the Edit pill renders blue (`#4ECBE5`) inside the Access Control cell when granted. Each expanded group row also shows a **`GroupCapabilitiesCard`** sourced from `GROUP_BEHAVIOR` (Sections / Pod axis / Client scope chips); the same data drives the `▸ How groups work` reference table at the top of the Groups tab. Keep `GROUP_BEHAVIOR` in sync with `_DEFAULT_PERMISSIONS` + resolver outputs in `backend/app/services/access.py`.
- **Preview-as flow**: `setPreviewAs(email, returnPath?)` saves the admin's current path before swapping the impersonation header. The global `<PreviewBanner>` (`src/components/layout/PreviewBanner.tsx`, mounted in `(app)/layout.tsx`) shows a sticky amber banner with an Exit button on every page. On entry, the launcher redirects to `firstAccessibleRoute(view_slugs)` so the admin lands on a page the previewed user can use; on exit, `consumePreviewReturnPath()` returns them to where they started. `mounted` guard avoids hydration mismatch on first paint.
- **Pod-axis toggle**: `useCurrentPodAxis()` (`src/lib/podAxisClient.tsx`) returns `{ axis, canToggle, setAxis }`. Charts that group by pod read `axis` and pick `c.editorial_pod` vs `c.growth_pod`. The toggle UI lives in `SyncControls` (animated segmented control via framer-motion `layoutId`; Editorial = green, Growth = sky-blue) and renders only when `canToggle` (Admin / VPs and Managers / BI Team) AND the current pathname is in `POD_AXIS_ROUTES` (Overview / Editorial Clients / Team KPIs). Hidden on Admin, Data Management, and Capacity Planning pages where it has no effect. Pod-locked teams (Editorial Team / Growth Team) always read their pod kind. **Every chip / label / aggregation that groups by pod must follow `axis`** — pass `podAxis` into `displayPod(pod, podAxis)` and `podBadge(pod, podAxis)`, and pick the right field in your aggregator. Reference implementations: `ContractClientProgress.usePodAggregates`, `DeliveryOverviewCards.PodAttentionCard`, `TimeToMetrics` (Client Milestone Journey + per-card popups + per-client trend), `CumulativePipelineSection`, `ClientDeliveryCards`, `ClientEngagementTimeline`. Empty-state copy reads "growth-pod" or "editorial-pod" depending on axis — keep that pattern.
- **Overview comments**: `OverviewCommentsRail.tsx` exports `<OverviewCommentsProvider>` (wraps the Overview content + supplies `filteredClients` via context) and `<SectionCommentIcon sectionId sectionLabel>` (drop into each Section's title row — Notion-style bare glyph, `MessageSquarePlus` for empty sections (hover-revealed), `MessageSquare` + amber count for sections with threads). Clicking the icon opens an anchored popover (no backdrop, no layout shift). Composer's client picker mirrors `FilterBar`'s typeahead "Search clients..." style. Timestamps use `formatRelative` — Notion convention (`now / Xm / Xh / 10:42 AM / Yesterday / May 8 / May 8, 2024`). To add a section that participates, append it to the `SECTIONS` array on the Overview page AND mount a `<SectionCommentIcon>` in that Section's `trailingSlot`.
- **`FilterBar` zero-match fallback**: when non-date filters yield no clients, the date range resets to "All Time" so the user isn't stuck with an empty window. Mirror this if you add new filter dimensions.
- **Tooltip body**: import `TooltipBody` from `shared-helpers.tsx` for every metric tooltip — uppercase mono title + 2–3 short bullets. Trigger should be tight (no large hit-area padding). `DataSourceBadge` is the only intentional exception (source metadata, not metric explanation).

## House rules

- Dark surfaces only (`bg-[#0d0d0d]` / `#161616` / `#1F1F1F`). Never use `bg-background` token for tooltips — it resolves unreliably and renders transparent.
- `DataSourceBadge` on every chart/section. The `source` prop carries a human description of the source sheet + formula.
- Emoji-free.
- No `Co-Authored-By` lines in commit messages.
