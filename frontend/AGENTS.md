## Project quick-reference

- **Routes + folder structure**: `README.md`
- **Auth flow**: `src/lib/auth.ts`, `src/lib/session.ts`, `proxy.ts`
- **API client**: `src/lib/api.ts` (`apiGet`/`apiPost` against `NEXT_PUBLIC_API_URL`)
- **Shared types**: `src/lib/types.ts` — mirror of `backend/app/schemas.py`. When you add or rename a field in either place, update both.
- **Store pattern (CP v2)**: `src/app/(app)/capacity-planning/_store.tsx` is a React context backed by `localStorage`. Every CP v2 page reads + writes through this store. When the backend ships `/api/cp2/*` endpoints, swap `localStorage` calls for `apiGet`/`apiPost`; UI stays the same.
- **Filter bar**: `src/components/dashboard/FilterBar.tsx` fans out filtered client list + date range via `onFilterChange` + `onDateRangeChange`. Dashboards that want time-aware charts must consume both. Status options: `All` / `Active` / `Soon to be active` (`SOON_TO_BE_ACTIVE`) / `Inactive/Completed`.
- **Deliverables pagination client**: `src/lib/deliverablesClient.ts` — `fetchAllDeliverables()` pages through `/api/deliverables/` in 1000-row batches until the API returns a short page. Use this instead of a bare `apiGet` with a fixed limit. Overview and Editorial Clients pages both call it.
- **Scroll/reveal helper**: `src/lib/detailTargets.ts` — `revealDetailTarget(id, kind)` opens any `<details>` ancestors, smooth-scrolls the element into view, and flashes a ring/outline highlight for 1.6 s. `revealCurrentHashTarget(prefixes)` applies it to the URL fragment on mount + `hashchange`. Used by `DeliveryOverviewCards` (client/pod click-throughs from Overview) and `CumulativePipelineSection` (hash deep-links from Overview).
- **Pod label normalization**: `normalizePod()` appears in several files (`FilterBar.tsx`, `ContractClientProgress.tsx`, `DeliveryTrendChart.tsx`, `CumulativePipelineHeader.tsx`). "Pod 1" / "1" / "pod 1" all collapse to "Pod 1". Keep the function in sync across files or extract to `src/lib/pods.ts` next time someone touches it.
- **Pod display formatter**: `displayPod(pod, "editorial" | "growth")` in `src/components/dashboard/shared-helpers.tsx`. Returns `"Editorial Pod 1"` / `"Growth Pod 1"` / `"Unassigned"`. Keys passed around the code (`"Pod 1"` etc.) are never shown verbatim in user-facing copy — always wrap in `displayPod()` when rendering. `podBadge()` already does this internally.
- **Sync UX**: `SyncControls` (`src/components/layout/SyncControls.tsx`) holds the SYNC button + the "Synced Apr 23, 4:18 PM" badge. It's rendered inline with the filter row on D1/D2 (since `Header` returns null on those routes) and inside `Header` for every other route. Click → `SyncAllModal` → per-sheet fan-out to `/api/migrate/import` → synthetic **"Refresh computed KPIs"** step calling `POST /api/migrate/refresh-kpis` (recomputes the four Notion-derived KPIs from `notion_articles` + `capacity_projections` for every month with source data, capped at 36 months back) → final render is `SyncResultDetail`. One `window.dispatchEvent(new Event("data-synced"))` fires at the end; dashboards + the badge listen for this to refetch. The badge reads from `GET /api/migrate/status` (audit_log row written on every successful import) and renders the timestamp via `toLocaleString()` so it's automatically locale + timezone aware. Backend stores naive UTC datetimes — the helper in `SyncControls` appends `Z` if missing before parsing so we never drift by the user's UTC offset.
- **Team KPIs filter bar (`TeamKpiFilterBar`)**: matches D1's UX. Date period uses the same `DateRangeFilter` component (calendar + month-range slider + presets) — no separate Month/Year dropdowns. Member + Client are typeahead comboboxes via the local `FilterCombobox` helper (text input filters the dropdown, click commits, `X` clears). The page state is `dateRange` (not `month/year`); `/api/kpis/` is fetched with `year_from / month_from / year_to / month_to` and the heatmap aggregates per `(member × kpi_type)` using mean of non-null scores + the latest non-null target. Hovering any KPI column header surfaces a `TooltipBody` with source / formula / target / direction / **caveats** (paused upstream for AI Compliance, fallback heuristic for Second Reviews, pod-level replication for Capacity Utilization, status-name dependency for Revision Rate). Tooltip content lives in `KPI_HEATMAP_TOOLTIPS` at the top of `team-kpis/page.tsx` — keep that table the source of truth for caveats.
- **Section-header style**: Every top-level section on Dashboard 1 uses the same h2 pattern — `font-mono text-base font-bold uppercase tracking-[0.2em] text-white` + `h-px flex-1 bg-[#2a2a2a]` horizontal rule on the right. Card h3s use `text-sm font-semibold` (`#C4BCAA`). When adding a new section, mirror this to keep the visual hierarchy intact.
- **Sticky section headers + anchor nav**: each top-level `<section>` on D1/D2 carries `id="…"` + `scroll-mt-[140px]`, and the h2 wrapper is `sticky top-[120px] z-10 bg-black pb-2 pt-1`. The filter+tabs band that owns the top of the page has `sticky top-0 z-20` + `min-h-[120px]` so its `bg-black` butts up against the h2 sticky position with no transparent gap. Each tab also renders `SectionIndex` (`src/components/dashboard/SectionIndex.tsx`) — a sticky left-side rail with scroll-spy + click-to-jump (`topOffset = 140` matches `scroll-mt`). Critical: the page's scroller is the `overflow-auto` `<div>` from `app/(app)/layout.tsx`, not `window`. `SectionIndex` walks up the DOM and listens to every scrollable ancestor — keep it that way.
- **Scope-aware overview cards**: the top of each Deliverables-vs-SOW section renders one of three card sets depending on filter scope — single client / single pod / portfolio. Implementations: `DeliveryOverviewCards`, `GoalsOverviewCards`. `DeliveryMixCard` (Delivery Progress) is the lead triage card for **every** scope, including single-client (scoped to that one client); the `RatioCard`s, `ClientStatusCard`, `ContractTimingCard`, and `ClosingSoonCard` round out each lineup. `MostBehindCard` and `PodAttentionCard` are exported from `DeliveryOverviewCards.tsx` but **not** mounted in D1 portfolio mode — they live exclusively on `/overview`. `framer-motion`'s `AnimatePresence` handles the layout swap when scope changes. Cards use `h-full` + `grid align-items: stretch` so the row aligns to the tallest card. The Cumulative Pipeline section instead uses an inline strip header (`CumulativePipelineHeader`): scope summary + 4 stage mini-bars + anomaly chip for portfolio/pod, status chip + pod chips + contract window for single-client. The per-client cards below carry the real detail.
- **Triage variance math (single source of truth)**: every "is this client behind?" calculation on /overview and the per-client cards uses **cumulative variance through end of the current contract Q** — `Σ delivered actuals + Σ remaining-month projections IN this Q − Σ contracted invoicing from start through end of this Q`. Catch-up over-delivery in one Q nets against earlier-Q deficits so a `−14 last Q + 14 this Q` reads as 0, mirroring the spreadsheet's Variance row byte-for-byte. Signed buckets (over-delivery is Healthy, NOT magnitude-based): `v ≥ 0` Healthy · `−5 ≤ v < 0` Within limit · `v < −5` Behind. Implementations: `computeCurrentQ()` + `clientCurrentQTier()` in `DeliveryOverviewCards.tsx` (Triage cards) and `quarterMetaFromPeriods()` in `ClientDeliveryCards.tsx` (per-client cards). The Monthly detail popover's "Variance" column also uses cumulative math — don't switch any of these surfaces to per-Q in isolation without updating the others.
- **"New (1st Q)" escape hatch**: clients whose current contract Q is their first (Q1 of year 1) always look behind on contracted invoicing because they haven't had time to ramp. Triage cards + `ClientDeliveryCard` classify them as a 4th tier (`new`, blue chip color `#8FB5D9`) instead of lumping them into Behind. They surface in a dedicated popover with their actual numbers (variance + projected end), just without alarm coloring. Check via `isFirstContractQ()` in `DeliveryOverviewCards.tsx` (consumes `SummaryRow`) and `qMeta.currentQ?.label === "Q1"` in `ClientDeliveryCards.tsx`.
- **Per-card tier badge** on every `ClientDeliveryCard`: a small pill next to the client name surfaces `HEALTHY` (green) / `WITHIN LIMIT` (amber) / `BEHIND` (red) / `1ST Q` (blue) — same thresholds as the Overview Delivery Progress card so both surfaces classify identically. Computed by `computeClientTier()` in `ClientDeliveryCards.tsx`; rendered via `<TierBadge>`. When `tier === "new"` the card also softens the projected-variance row so brand-new clients never read as `Behind plan`.
- **"End-of-Q variance" note** on every `ClientDeliveryCards` card: the line below the Current Q progress bar shows the cumulative projected variance (delivered − invoiced through end of current Q) + a copy hint (`On track` / `Slight drift` / `Behind plan` / `New 1st Q`). The number is colored by signed variance (green ≥ 0 · amber −5 to 0 · red < −5). The eyebrow reads `END-OF-Q VARIANCE` and the value carries an `articles` unit chip so it's unambiguous what the number represents. Component: `ProjectedEndOfQNote` in `ClientDeliveryCards.tsx`.
- **Variance wording is canonical across the Triage stack**: every headline, subtitle, tooltip, and popover on `DeliveryMixCard` / `MostBehindCard` / `PodAttentionCard` calls out *projected end-of-current-Q variance* explicitly. The formula `delivered − invoiced` appears in at least one bullet per card so anyone hovering knows what −5 / −15 etc. represent. Don't fall back to vague phrasings like "projected close" — use "projected end-of-Q variance".
- **Overview dashboard (`/overview`)**: exec snapshot. THREE sections, all driven by the same `FilterBar`: `<PeriodSnapshotSection>` (Pod Snapshot — `<PodDeliveryProgressCard>` per-pod + per-client Goals / Last Q / Current Q / %SOW / %Published), `<PodPaceSection>` (Time to Milestones — three cards stacked: `<PodMilestoneJourneyCard>` (Pod Timelines), `<PodTTMStatsCard>` (Time-to-Metrics), and `<TimeToTrendChart>` (Per-Client Days)), and `<ProductionTrendChart>` (Production History, see *Production History chart* below). `SECTIONS` (the left nav rail) lists `period-snapshot` · `pod-pace` · `production-history`. Comments live per-section via `<OverviewCommentsProvider>` + `<SectionCommentIcon trailingSlot>` inline next to each section title. Lifetime delivery summaries come from `buildLifetimeSummaries()` in `src/lib/overviewSummary.ts`. The "Link cards" toggle (rendered in the Time to Milestones section's `rightSlot`) governs cross-card hover + click-to-focus effects, persisted in `localStorage` (`pod-pace-link-enabled`). Page-level gate via `useRequireView("overview")` redirects pod-locked teams to `/editorial-clients`.
- **Pod Snapshot drill-downs (`ClientDetailPopover`)**: every cell in `PeriodSnapshotSection`'s per-client row opens a click-anchored popover with five variants — `client` / `goals` / `lastQ` / `currentQ` / `lifetime`. The `client` variant is INFO-ONLY (status + pod chips + `ContractMetaBlock`) — the Goals / Q / Lifetime blocks live in their own popover variants triggered from the respective cells. The `currentQ` variant leads with the End-of-Q variance + tier, two lifetime bars (Q delivered, Invoiced) sized from ACTUAL delivered to date, monthly breakdown table (variance cells tinted by tier), and an AS-OF anchored SOW progress chart that shows the full contract span (trims leading zero months and centres on the AS-OF cell). Popover closes on outside-click, Esc, OR scroll on any scrollable ancestor (capture-phase listener; scrolls inside the popover body don't trigger close).
- **Current Q semantics — actual delivered vs projected variance**: in every Current Q surface (Pod Snapshot pod-summary tile via `QTile`, per-client cell via `ClientQCell`, popover `QSummaryBars`, `ClientDeliveryCards.QuarterRow`), the progress BAR's numerator is **actual cumulative delivered through the last completed month** (not projected end-of-Q) while the denominator stays cumulative invoiced through end-of-Q. The variance NUMBER + tier come from the projected end-of-Q outcome (`projectedEnd − cumInvoiced`). Pod-aggregate Current Q totals include 1st-Q clients in the bar (their delivered is real); only the pod-level variance + pace-weight metrics skip 1st-Q clients. The Client Delivery card's "Lifetime · SOW" row uses the same shape — Delivered + Invoiced bars compute their numerators from `row.monthly_breakdown` (`Σ delivered / invoiced where !is_future`), NOT the date-filter-scoped `row.articles_delivered` / `row.articles_invoiced` which would distort under a date filter. Keep all five surfaces consistent when the semantic changes again.
- **Cross-card link in Pod Pace section**: a single `LinkedHover` state in `PodPaceSection` connects three cards via the "Link cards" toggle. Hover a Time-to-Metrics card OR a Pod Timelines segment / dot → matching elements glow in the other cards, non-matching dim, and `<TimeToTrendChart>` receives a `metricOverride` so the Per-Client Days dropdown transiently switches to the hovered transition. Click a per-client row in Pod Timelines → `selectedClientId` re-scopes `PodTTMStatsCard` (cards now describe that one client's days), the row gets a green ring, and `<TimeToTrendChart highlightedClientName>` rings that client's bar while dimming the others. Click the same row again to clear. Toggle off → all cross-card effects stop.
- **Milestone numbering**: helpers `MILESTONE_NUM_BY_FIELD` + `milestonePairPrefix(fromField, toField)` in `shared-helpers.tsx` drive the numbered prefixes for the six milestones (`1` Consulting KO · `2` Editorial KO · `3` First CB Approved · `4` First Article · `5` First Feedback · `6` First Published). Surfaces using the prefix: Pod Timelines legend (`<LegendChip>`), `<TTMStatCard>` titles (e.g. `1→2 · CKO → EKO`), `TimeToTrendChart`'s metric dropdown options, the JourneyTooltip's focal stat + "after previous milestone" leg labels, and `TimeToMetrics.HoverableMetricCard` titles. When you add a new milestone, register it in `MILESTONE_NUM_BY_FIELD` and update both `TTM_METRICS` (Pod Pace section) and `METRIC_DEFS` / `JOURNEY` (legacy TimeToMetrics).
- **Production History chart (`<ProductionTrendChart>`)**: three view modes — `All` (combined Actual + Projected line with gradient fill), `Per pod` (one line per pod, no fill), `Per client` (one line per client — **only available when filter narrows to one pod**, otherwise hidden via `allowPerClient`). Tooltip is rendered OUTSIDE Recharts as `<ChartTooltipBox>` with `position: fixed` + viewport-clamped Y, anchored to the data column (not the cursor), opposite side of the cursor's chart half so the active month stays visible. `<TooltipProbe>` observes Recharts' active/payload/label and reports label changes via `useEffect` so the box re-anchors only when the active month changes (not on every mousemove). All three modes feed `breakdown[]` / `breakdownByPod[]` arrays on each `ChartRow` for the tooltip; per-client mode also bridges Actual → Projected per series so dashed/solid lines connect across the NOW line. Colours pull from `POD_HEX_COLORS` in `shared-helpers.tsx`.
- **Cumulative Pipeline — Articles uses `articles_sent`**: Topics and CBs use the `approved` field from the Cumulative sheet because those gates require explicit client sign-off. Articles uses `articles_sent` because articles delivered to the client are billable regardless of the client's downstream approval workflow. Published uses `published_live`. Applies in `CumulativePipelineHeader.tsx` (the strip header on D1 + `/overview`) and `ClientPipelineCard.tsx` (the per-client cards below). - **Pipeline stage palette**: `PIPELINE_STAGE_COLORS` in `src/components/dashboard/shared-helpers.tsx` — Topics #2E8C59 → CBs #42CA80 → Articles #65FFAA → Published #DDCFAC. Strictly Graphite DS swatches (P3 → P2 → P1 + WN1). Don't reintroduce S1/S2/S8 (blue/purple/cyan) on pipeline bars — flagged off-brand.
- **Pacing-aware coloring**: percentages that mean "% of contract progress" (e.g. lifetime delivered) use `pacingColor(actualPct, elapsedPct)` from `shared-helpers.tsx` — green inside ±10pp of expected, yellow at -25pp, red beyond. New clients (<8 % elapsed) always read green. Don't fall back to raw 75 / 50 thresholds for these metrics.
- **Content-type weighting**: goals aggregations multiply CBs / articles by `contentTypeRatio()` (article ×1, jumbo ×2, LP ×0.5). The content-type table is authoritative; the sheet's `ratios` column is fallback only because direction (`1:2` vs `2:1`) was inconsistent. - **`FilterBar` zero-match fallback**: when non-date filters yield no clients, the date range resets to "All Time" so the user isn't stuck with an empty window. Mirror this if you add new filter dimensions.
- **Tooltip body**: import `TooltipBody` from `shared-helpers.tsx` for every metric tooltip — uppercase mono title + 2–3 short bullets. Trigger should be tight (no large hit-area padding). **`DataSourceBadge` is now a no-op** (returns null) so existing call sites compile but render nothing — the LIVE / source chip was deemed visual noise across the Hub.
- **Tooltip copy style** (applies to every dashboard surface — Overview, D1, D2, shared cards, charts, Admin pills). Audience is non-technical operators. Strict rules:
  1. Max **3 bullets** per tooltip. Title ≤ 4 words, sentence case.
  2. Each bullet ≤ **10 words**. One short declarative thought, terminal period.
  3. **No code, table, or sheet-tab names** in user-facing strings.
  4. Math is fine, plain symbols: `delivered ÷ invoiced`, `Σ`, `±5`. Don't write out "variance" jargon — say "behind / ahead".
  5. End with an action when the row is clickable: `Click a row to jump.`
  6. CP2 sub-pages are an explicit exception while they're [Proposal] — their tooltips can keep schema references (`cp2_fact_*`) until that subsystem ships to end users.

## House rules

- Dark surfaces only (`bg-[#0d0d0d]` / `#161616` / `#1F1F1F`). Never use `bg-background` token for tooltips — it resolves unreliably and renders transparent.
- `DataSourceBadge` is a no-op as of 0.3.15 — don't add new instances; existing ones render null.
- Section header style: `font-mono text-sm font-semibold uppercase tracking-widest text-[#C4BCAA]`. Don't reintroduce the older sticky `top-[120px]` + `border-b` + horizontal-rule pattern; section h2s now read as plain text rows on every dashboard.
- Emoji-free.
- No `Co-Authored-By` lines in commit messages.
